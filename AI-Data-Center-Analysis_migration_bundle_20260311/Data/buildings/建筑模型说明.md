# DRL_DC 建筑模型说明 (清理后)

本文档描述清理后的 `DRL_DC_training.epJSON` 和 `DRL_DC_evaluation.epJSON` 模型。

---

## 1. PlantLoop 拓扑

模型包含 **2 个水环路**:

### Chilled Water Loop (冷冻水环路)
- **供给侧**: Chiller (实际 `reference_capacity=50 MW`, COP=6.28；对象名中的 `1230tons` 是历史命名，不作为当前容量口径) + Waterside Economizer (水侧自然冷却换热器) + TES Tank (Use Side, 放冷) + Bypass
- **需求侧**: CRAH Cooling Coil (机房空调冷却盘管) + Secondary Pump (变频二次泵) + TES Tank (Source Side, 充冷) + Bypass
- **控制**: CommonPipe, SingleSetpoint, Optimal load distribution
- **温度范围**: 1-40 C

**M1 新增: TES (蓄冷水罐) 单 loop 并联接入** — 见第 9 节。

### Condenser Water Loop (冷却水环路)
- **供给侧**: Cooling Tower (冷却塔) + Constant Speed Pump + Bypass
- **需求侧**: Chiller condenser side + Waterside Economizer + Bypass
- **控制**: SingleSetpoint, Optimal load distribution
- **温度范围**: 5-80 C

## 2. EMS Programs (当前核心 5 个)

### P_1 -- HVAC 控制程序 (12 行，纯滑动窗口)
根据 DRL agent 输出的归一化动作值，映射到实际设备控制值:
- **CRAH_T**: 送风温度 (11-22 C)，带 +/-0.5 C 步进约束
- **CRAH_Fan**: 风机风量 (500-2000 kg/s)，带 +/-100 步进约束
- **Chiller_T**: 冷冻水设定温度 (5.6-10 C)，带 +/-0.5 C 步进约束
- **CT_Pump**: 冷却水泵流量 (300-3300 kg/s)，带 +/-100 步进约束
- **当前 M2-F1 agent action stack 为 4 维**：`[CT_Pump_DRL, CRAH_T_DRL, Chiller_T_DRL, TES_DRL]`。底层 EnergyPlus 物理 actuator 仍是 5 维 full action `[CRAH_Fan_DRL, CT_Pump_DRL, CRAH_T_DRL, Chiller_T_DRL, TES_DRL]`，其中 `CRAH_Fan_DRL` 由 wrapper 固定插入为 1.0（安全满风量），不是 agent 动作；完整 HVAC 联合控制后移。
- `Chiller_Pump` actuator / sensor / schedule 仍保留在 epJSON 中作为历史接口，但 P_1 当前没有写入该 actuator。
- **无安全保护**: E0.1 修复中删除了 >30C 紧急保护（原因：41.8MW 过冷导致 1 步降温 15°C，比超温更有害）

### P_2 -- IT 负载程序
- **Training / Evaluation 版**: `SET ITE_rate = ITE_S`，使用 `ITE_Set` 的 schedule 值作为 IT 负载率。
- **当前 M2 约定**: `ITE_Set` 保留为外部场景/探针写入入口，但不再是 agent 动作维度；默认值 0.45 表示固定 IT 负载率。

### P_5 / P_6 / P_7 -- TES 与 Chiller 守护程序
- **P_5** (`BeginTimestepBeforePredictor`): 读取 `TES_Set`，控制 TES Use/Source availability 与流量上限，并在放冷且 SOC>0.15 时关停 Chiller。
- **P_6** (`EndOfZoneTimestepAfterZoneReporting`): 根据 10 个水罐节点温度计算 SOC 与平均罐温，并写入观测 Schedule。
- **P_7** (`InsideHVACSystemIterationLoop`): 在 HVAC 迭代内部重复写入 TES 流量和 Chiller 关停 actuator，避免 solver 覆盖 P_5 的设置。

### 9.2 接入拓扑 (方案 A — 单 Chilled Water Loop 并联)

```
Supply 侧 (冷源):
  Inlet Branch → Splitter → {Branch 1: Chiller,
                              Branch 2: Chiller Bypass,
                              Branch 3: TES Use Side (放冷)} → Mixer → Outlet Branch

Demand 侧 (冷汇):
  Inlet Branch → Splitter → {Branch 1: Coil Bypass,
                              Branch 2: CRAH Coil,
                              Branch 3: TES Source Side (充冷),
                              Bypass Branch} → Mixer → Outlet Branch
```

- **TES Use Side** 挂在 Supply Branch 3，作为制冷**供应方**（放冷给回路）
- **TES Source Side** 挂在 Demand Branch 3，作为制冷**消耗方**（从 chiller 充冷）
- 不添加独立泵 — 复用已有 Primary Pump (Constant) + Secondary Pump (VariableSpeed)
- TES 已加入 `Chilled Water Loop Cooling Equipment List` 作为第 3 个 equipment

### 9.3 节点命名

| 节点 | 物理含义 |
|---|---|
| `CW Tank Use Inlet Node` | Use side 进口（放冷时接收回路回水） |
| `CW Tank Use Outlet Node` | Use side 出口（放冷时向供水侧输出冷水） |
| `CW Tank Source Inlet Node` | Source side 进口（充冷时接收 chiller 冷水） |
| `CW Tank Source Outlet Node` | Source side 出口（充冷后回到需求侧回路） |

### 9.4 新增 Output:Variable (编号 16/17/18/20)

**EMS:Program**（2026-04-23 已修正方向）：

- **P_5** — TES 充放模式控制（CallingPoint: `BeginTimestepBeforePredictor`）
  - 读 TES_Set_Sensor、SOC
  - `IF TES_Signal > +0.01`（放电分支）：Use_Avail=1, Use_MFlow=|v|*97.2
  - `ELSEIF TES_Signal < -0.01`（充电分支）：Source_Avail=1, Source_MFlow=|v|*97.2
  - 默认 Chiller ON（Chiller_Branch_Avail=1, Chiller_Out_T_SP=6.67, Chiller_In_MFlow_Max=9999）
  - **放电模式叠加关 Chiller**：`IF TES_Signal > +0.01 AND SOC > 0.15` 时，4 个 actuator 设为关停（Chiller_Branch_Avail=0, Component_Avail=0, Out_T_SP=30, In_MFlow_Max=0）
  - SOC ≤ 0.15 自动恢复 Chiller（与 reward.soc_low=0.15 对齐）
- **P_6** — SOC 计算（CallingPoint: `EndOfZoneTimestepAfterZoneReporting`）
  - T_tank_avg = T_tank
  - SOC = (12°C - T_tank_avg) / (12°C - 6.67°C)，Clip 到 [0, 1]
  - 2026-04-28 修正：冷端从旧 6.0°C 对齐到 P_5 / Chiller 当前实际供冷 setpoint 6.67°C，避免水罐温度已到物理可达冷端时 SOC 仍停在约 0.888。
  - 写入 TES_SOC_Obs 和 TES_Avg_Temp_Obs
- **P_7** — Chiller 关停 HVAC 迭代守护（CallingPoint: `InsideHVACSystemIterationLoop`）
  - **非冗余**：BeginTimestepBeforePredictor 的 actuator 设置在 HVAC solver 迭代中会被内部控制器覆盖，需要此守护反复强制
  - 条件：`IF TES_Set_Sensor > +0.01 AND SOC > 0.15`（即放电分支省电时），在 HVAC 迭代内把 4 个 Chiller actuator 重设为关停

**关键设计决策**：

1. **4 种 actuator 叠加关 Chiller**：Branch On/Off / Component On/Off / Outlet T Setpoint / Inlet MFlow Max 单独任何一个都无法彻底关停 Chiller（测试中单独用只能降到 ~1.3 MW），必须**四管齐下 + HVAC 迭代循环强制**才能真正关停
2. **TES_Set 使用无边界 `Any Number` schedule type limit**：之前用 Fractional 会让 -0.5 被 E+ 裁剪为 0 进而完全失效，现在仅声明连续数值类型、不设置上下界（仅 EMS P_5 逻辑读取）
3. **P_6 的 SOC 初值**：仿真启动时 TES_SOC_Obs=0.5（schedule 默认值），首个 timestep 后写入真实 SOC。Mixed 水罐的初始热状态由 EnergyPlus 在环境初始化时按 `TES_Charge_Setpoint=6.0°C` 建立；SOC 观测标尺以 6.67°C 作为实际可达冷端；当前不在 wrapper 中伪造随机初始 SOC。

### 9.13 2026-04-25 诊断记录：Max_Flow 试错与真实 root cause 定位

**初始假设（已证伪）**：phased valve→SOC probe（`tools/m1/probe_valve_soc.py`，Nanjing Jan 1-7，15-min timestep）在 §9.12 P_5/P_7 方向修复后做物理性能复测，发现放/充电方向已正确但放冷峰值功率严重不足（Phase 1 放电期 Zone `air_temp` 峰值 **40.24°C**），第一时间假设 P_5 第 2 行 `SET Max_Flow = 97.2`（kg/s）数值过小（折合 2.44 MW，只到名义放冷功率 9.77 MW 的 1/4）是 root cause，将其改为 `389.0` kg/s（对齐 9.77 MW × ΔT=6°C × cp=4.186）。

**Sensitivity test 结论（证伪）**：随后将 `Max_Flow` 在 50 / 389 / 1000 三个数量级反复切换并各跑一次 1-week probe，所得 `eplusout.csv` 与 `monitor.csv` **逐字节相同**。说明在 plant 路由问题未解决前，`Max_Flow` 这个常量不会传播到任何水力学输出 → "97.2 → 389" 是 cosmetic 修改，不构成修复。

**Phased probe 真实现象**：
- Phase 1（v=+0.5 放电）前 61 h `use_HT = 0`，Zone `air_temp` 持续爬升至 40°C；
- 第 61 h 后 4 个 timestep 突发倾泻 13.6 MW（远超名义放冷功率 9.77 MW）→ plant 收敛阈值触发的"积压排空"，不是受控放冷。

**修复 4（追加，2026-04-25 后续诊断）：Chiller Bypass 关闭 + 季节性验证**

修复 1-3 完成后初步看到 winter probe 中 use_HT max 仅 1.85 MW（远低于名义放冷功率 9.77 MW），最初怀疑 Bypass 分流——遂加 EMS:Actuator `Chiller_Bypass_Avail` (`Plant Component Pipe:Adiabatic / On/Off Supervisory`)，在 P_5 / P_7 chiller-kill 块叠加：

**季节性 probe 验证**（确认 winter 1.85 MW 不是 bug 是物理）：
