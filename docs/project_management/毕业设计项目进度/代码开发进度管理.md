## 1. 项目目标

项目名称：面向光伏自消纳的 AI 数据中心冷却-蓄冷-算力联合优化系统设计

核心研究问题：reward shaping 如何影响 TES 蓄冷和弹性 IT 负荷的调度方向，对比 RL-Cost 与 RL-Green 两种 reward 下 agent 行为差异。

### 5.x.2 E0.2 Reward 函数修正（2026-04-05）

E0.1 修复后启动 12 seed × 300ep 训练，ep200 时发现大面积训练发散（12 个 seed 中 5 个 DIVERGED、4 个 UNSTABLE、仅 3 个 HEALTHY）。

**问题诊断过程**：
1. 部署了 metrics_recorder.py 每 120s 采样 ent_coef、critic_loss、reward、comfort
2. 收集 816 个数据点（12 seed × 68 采样）后做相关性和时序分析
3. 发现所有 seed（包括健康的）都在 ep61-80 遇到过极端 worst reward（-46000 ~ -53000），是正常 reward（-6000）的 8 倍
4. 健康 seed（05, 06, 12）在极端事件后恢复，发散 seed 没恢复 → 区别不在于是否遇到极端事件，而在于 critic 网络能否消化极端 reward 幅度
5. comfort violation 与 reward 的相关性最强（r=-0.881），ent_coef 飙升是症状而非病因

**根因**：线性 comfort 惩罚在极端超温时产生过大的 reward 信号（40°C 时惩罚是正常的 8 倍），击穿部分 seed 的 critic 网络，触发 ent_coef 飙升的恶性循环。

**讨论过的方案**：
- 恢复渐进保护（27/28/29/30°C 四级 EMS 干预）→ 否决：限制 agent 探索空间、引入人为偏差、论文难解释
- 固定 ent_coef=0.02 → 否决：只切断恶性循环的放大器，不治根因，且 ent_coef 飙升被证明是症状不是病因
- Reward clipping（封顶惩罚）→ 否决：硬截断不优雅，agent 无法区分 35°C 和 45°C
- 非线性惩罚 + 收紧温度区间 → **采纳**

**最终修改**：

| 编号 | 修改 | 理由 | 文件 |
|------|------|------|------|
| E0.2-01 | comfort 惩罚 线性 → sqrt（平方根） | sqrt 保留"越高越差"的单调梯度，但压缩极端值幅度（线性 10°C→罚 10，sqrt→罚 3.16），防止 critic 被极端值击穿 | sinergym/utils/rewards.py PUE_Reward._get_comfort_penalty |
| E0.2-02 | 舒适温度上限 27°C → 25°C | 收紧到 ASHRAE A1 推荐值，更早开始产生惩罚信号，让 agent 在温和区间就学会温控，减少冲到 30°C+ 的概率 | sinergym/__init__.py reward_kwargs |

### 5.x.3 E0.3 DSAC-T 替代标准 SAC（2026-04-13 ~ 04-16）

E0.2 重训仍然大面积发散（12 seed 中仅 2 个健康）。文献调研（Ali et al. 2021 描述 SAC "熵过估计 → alpha 升高 → 更多随机 → 更多熵过估计"正反馈环）+ SSAC 论文（MDPI Buildings 2025）确认 **SAC 自动熵调节是 HVAC RL 的已知不稳定来源**。

**决策**：实现 DSAC-T（Distributional SAC with Three Refinements, arXiv:2310.05858），严格按论文公式复现。

**核心改进**：
- R1 **Expected Value Substituting**：用确定性 target Q 替代随机采样，减少梯度噪声
- R2 **Twin Value Distribution Learning**：两个分布式 Critic 取均值较小者（防过估计）
- R3 **Variance-Based Critic Gradient Adjustment**：方差自适应梯度缩放——高不确定性区域学得慢，抑制极端 reward 击穿 critic

**实现文件**：
- `tools/distributional_critic.py` - 分布式 Critic（输出均值 + 标准差）
- `tools/dsac_t.py` - DSAC-T 算法类（继承 SAC，重写 train()）

### 5.x.1 E0 数据验证发现的问题详述（2026-04-03）

E0 基线训练（4 seeds × 120 episodes）完成后，对 seed02-best 模型做了全年 8760 小时逐时数据验证。以下是发现的问题、分析过程和决策：

#### 问题 1：训练/评估 IT 负载分布不一致

| | 训练 (P_2 随机游走) | 评估 (Earth trace 注入) |
|---|---|---|
| 范围 | [0.25, 0.75] | [0.021, 0.767] |
| 最低负载 | 25%（~3745 kW）| 2%（~300 kW）|
| 时序结构 | 无记忆布朗运动 | 有日夜/周间周期（lag1=0.985, lag24=0.864）|

**影响**：agent 从未见过 IT < 25% 的工况（评估中有 217 小时低于 20%），冷却策略在低负载时段不合理。

**讨论过的方案**：
- 扩展随机游走范围（选定）
- 训练时也注入真实 trace（7 条可用：Earth/Kalos/PAI2020/Saturn/Seren/Uranus/Venus）
- 在 P_2 中加入正弦日内周期

**决策**：E0 阶段只扩展范围到 [0.05, 0.80]，保持随机游走。理由：E0 目标是验证模型可靠性，不引入 SustainDC 的 workload 模块。时序结构差异是已知限制，在论文中作为 limitation 讨论。

#### 问题 2：EMS 紧急保护导致物理不可能的温度骤降

**现象**：室温 > 30°C 时触发紧急保护，1 步内降温 15.1°C（30.8°C → 15.7°C）。全年 5 次触发（1月1次、2月3次、11月1次），全部在冬季/过渡季。

**根因分析**：
- 紧急保护设定：Fan=2000 kg/s, 送风=10°C → 冷却功率 41.8 MW
- IT 发热量仅 ~7.5 MW → 过冷比 5.6:1
- epJSON 无 InternalMass 对象，无 ZoneCapacitanceMultiplier
- **但核心问题不在热容**：即使加 20 倍热容量乘数，41.8 MW 净冷却仍能在 1 小时内大幅降温。问题在于 HVAC 超配（Fan 2000 kg/s 是实际需求 750 kg/s 的 2.7 倍）+ 送风温度被压到 10°C

**讨论过的方案**：
- 渐进保护（27/28/29/30°C 四级）
- 添加 ZoneCapacitanceMultiplier 增加热容（治标不治本）
- 直接删除保护

**决策**：直接删除。理由：
1. 保护机制本身制造了比超温更严重的过冷问题
2. 滑动窗口已提供安全约束（每步 ±100 kg/s / ±0.5°C）
3. agent 通过 comfort 惩罚学习温控
4. EnergyPlus 在 35-40°C 范围内数值稳定，不会崩溃

#### 问题 3：CT 水泵动作空间浪费

EMS 上限 5000 kg/s，设备 Autosize 容量 ~3259 kg/s。action [0.65, 1.0] 映射到同一物理值，35% 动作空间分辨率被浪费。改为 3300 kg/s。

#### 其他验证结论（正常，无需修改）

- **南京气象数据**：全年均值 16.9°C，1月 4.9°C，7月 28.0°C，范围 [-7, 38]°C — 与典型气候一致
- **IT 功率时滞**：EnergyPlus IT 功率 power[t] = f(util[t-1])，lag1 相关系数 0.9999。这是 EnergyPlus 标准行为，非 bug
- **功率平衡**：总用电 = IT + 冷却，全年最大偏差 0.01 kW — 数据提取管道无精度损失
- **冷水机出水温度**：与外温相关系数 -0.63，季节趋势正确（冬季 10°C，夏季 5.6°C）
- **制冷 vs IT 功率相关系数 0.53**：偏低（预期 >0.8），原因是 1 步时滞 + 滑动窗口限制响应速度
- **湿度 < 20%**：2069 小时。模型无加湿器，论文中作为 limitation 提及

### 6.2 容量选型的文献依据

**工业设计标准：4h × 峰值冷却负载**

引用自 [Azura Consultancy](https://www.azuraconsultancy.com/wise-efficient-use-of-thermal-energy-storage-tanks-in-data-centers/)：

> "if a data center with IT load of 4,000 kW would typically require 5,200 to 5,600 kW (1.3-1.4× IT load) of cooling capacity and hence the thermal storage capacity should be 4 hrs × 5,600 kW = 22,400 kWh or 6,370 Ton-Hr"

**应用到本项目**：
- IT 峰值：7.5 MW（热负载 = 电负载）
- TES 容量 = 4h × 7.5 MW = **30 MWh 热量 = 8,530 Ton-Hr**
- 水罐体积（ΔT=6K，T 冷=6°C，T 热=12°C）= 30,000 kWh / 6.98 kWh/m³ = **~4,300 m³**

**工程合理性对比**：
- Meta Clonee（爱尔兰数据中心）：7,570 m³
- NIH Bethesda：17,000 m³
- 本项目 4,300 m³ 在同一数量级，不超设计常识

**为什么不选 6h / 8h**：
- 6h 需要 6,450 m³、8h 需要 8,600 m³，进入超大型数据中心范围，经济性需额外论证
- 4h 对比传统 "10-15 min 备份容量"已是巨大提升，"削峰填谷"故事足够有力

**参考文献**：
- [Azura Consultancy - Wise & Efficient Use of TES Tanks in Data Centers](https://www.azuraconsultancy.com/wise-efficient-use-of-thermal-energy-storage-tanks-in-data-centers/) — **工业设计指南**，给出 4h × 峰值的标准 sizing 公式
- [MDPI Energies 2026 - Energy Storage Systems for AI Data Centers Review](https://www.mdpi.com/1996-1073/19/3/634) — 综述文章
- [ScienceDirect - Techno-economic performance of reservoir TES for DC cooling](https://www.sciencedirect.com/science/article/pii/S0306261925005884) — 5 MW DC 案例，LCO cooling $5/MWh vs 传统 $15/MWh
- [arXiv 2410.19830 - Peak Shaving using TES in CHP/District Energy](https://arxiv.org/abs/2410.19830) — 45,000 ton 校园级案例（UT Austin）
- [Reliable Resources - Thermal Storage for Data Centers](https://www.relres.com/news/79/thermal-storage-for-data-centers) — 技术概述与设计考量

**SOC 计算**（EMS Program）：
```
SOC = (T_hot - T_avg_nodes) / (T_hot - T_cold)
# T_hot = 12°C (完全放电), T_cold = 6°C (完全充电)
# T_avg_nodes = 节点温度按体积加权平均
```

### 7.7 M2-F1 TES 纯 RL 路线阶段性失败与路线调整（2026-05-02）

- M2-F1 主线使用无 workload action、fixed fan、trainlike `ITE_Set=0.45` 的 TES 调度基线，主线 observation/action 维度为 32/4。
- C2b 是 `34 obs / 4 action` 的 state augmentation 分支，用于显式暴露价差信号，但不增加动作维度。
- P1a 是 `34 obs / 5 action` + `direction_amp_hold` + `rule_tou` TES-head BC warm-start 分支，用于同时测试价差信号暴露、TES 方向保持动作和模仿初始化。

**已尝试路线**：
- reward / 探索侧：sparse PBRS、低价窗口扩展、entropy ablation。
- 训练组织侧：teacher/curriculum。
- 状态与动作侧：state augmentation、`direction_amp`、`direction_amp_hold`。
- 模仿初始化侧：`rule_tou` TES-head BC warm-start。

**关键结果**：
- C2b 在 96-step 短评估中曾出现改善，但 672-step 未保持。672-step 下 `mean charge_window_sign_rate≈0.306`，`low_price_discharge_fraction≈0.634`，说明短窗口收益没有转化为稳定跨日库存调度。
- P1a BC warm-start 5ep（run-115/run-116）中，eval/runtime 不启用 runtime planner 或 BC 控制；BC 只作为训练 warm-start / imitation learning 信号。672-step 评估结果为 `mean charge_window_sign_rate≈0.103`、`low_price_discharge_fraction≈0.655`、`low_price_valve_mean≈+0.323`、`high_price_valve_mean≈+0.376`、`comfort≈1.263%`、`guard≈0`。
- 阶段性结论是：高价时放冷相对容易学到；低价时主动充冷没有稳定学会。当前纯 RL 路线更像学到了即时放冷响应，而不是可靠的峰谷库存调度策略。

**经验教训**：
- `price_response` 不能单独作为成功指标。后续 TES 评估必须同时看低价充冷 sign-rate、SOC prepeak 增量、peak SOC 下降，以及 counterfactual / marginal saving。
- 短视 RL 在当前 reward、动作门控和训练预算下，倾向于学习即时放冷，而不是跨时段持有库存等待高价时段释放。
- 继续围绕 reward 系数、entropy、action scaling 做小幅调参的信息增益较低，难以证明路线能从根本上解决低价主动充冷问题。

**路线决策**：
- 准备放弃“纯 RL 自发探索 TES 峰谷套利”作为主路线，转向 planner/MPC oracle、可审计标签、BC/IL warm-start 或混合控制。
- 该调整不等于修改 EnergyPlus 物理，也不等于在最终运行时写死规则；目标是先用可审计 oracle 验证物理与 gate 下的可达上界，再把可达策略蒸馏进无 runtime planner/BC 的最终 policy。

**下一步建议**：
- 优先实现 TES-only MPC/planner oracle baseline，验证当前 TES 物理、EMS gate 和 comfort 约束下是否存在足够的峰谷套利上界。
- oracle 过门后，再做 MPC-label TES-head BC，并将其作为 warm-start / imitation learning 输入。
- 最终 eval 仍要求不启用 runtime planner / BC runtime 控制，即只评估训练后的 policy 行为；BC 只允许作为训练阶段的 warm-start / imitation learning 信号。

## 13. 论文叙事框架（2026-05-03 修订：MPC 主线 + RL 对照）

### 13.1 主贡献

构建可复现的 EnergyPlus + Sinergym 数据中心 TES 场景（Nanjing + 江苏 2025 TOU 合成 + Nanjing PVGIS 6 MWp + 1400 m³ 分层水罐 + EMS workaround + `TESTargetValveWrapper`），并给出**可审计的 supervisory MPC-lite 控制器**。在固定非 TES action / fixed fan / trainlike `ITE_Set=0.45` 条件下，MPC oracle 和 MPC-lite 双双通过 672-step mechanism gate（comfort 0.89%、charge_window_sign_rate 1.0、low_price_discharge_fraction 0、Δ_SOC_prepeak +0.25、Δ_SOC_peak −0.18、PUE 1.2596），证明 trainlike EnergyPlus/TES plant 在显式 anticipatory 控制器下能执行 TES 库存峰谷行为，**mechanism 物理可达**。

### 13.2 辅助贡献

纯 RL 失败分支系统性复盘：M2-F1 阶段 C1（reward shaping）、C2/C2b（state augmentation）、C3（entropy ablation）、A-prime（teacher curriculum）、O1（direction_amp_hold）、P1a（`rule_tou` BC warm-start）共七轮机制实验在相同物理与 wrapper 下都未能稳定学到"低价峰前主动充冷"。诊断结论：

- 高价时即时放冷（reactive control）相对容易学到；
- 低价时主动充冷（anticipatory inventory control）需要跨日 credit assignment，是 SAC/DSAC-T 在当前 reward / 动作门控 / 训练预算下的硬伤；
- 问题不在 EnergyPlus 物理、EMS 符号、阀门限速或 SOC guard，而在 learning。

由此阐述为什么显式 anticipatory MPC 是 TES 库存问题的合理 baseline，对应 §3 / §11 中 MPC 与 RL 实验的可达性差距。

### 13.3 可选贡献

MPC oracle monitor 生成 TES-head BC/IL warm-start labels（M2-G1），在 RL 训练阶段提供监督信号（M2-G2），最终 learned policy 评估时**不调用 runtime planner**，要求 BC/runtime 权重为 0（M2-G3）。该路线作为论文 Chapter "从 oracle 到无 planner learned policy"的进阶研究。

### 13.4 多站点 case study（OPTIONAL）

时间余量充裕时，展示方法从市场化电价（CAISO 历史失败，详见 §12）到 ToU 机制（江苏 / 安徽 / 宁夏）的迁移能力。Chapter 5 可包含 **CAISO 重尾 reward 失败的"算法边界"案例**，作为 DSAC-T 在批发市场重尾分布下稳定性挑战的开放性问题报告。三站点（南京 / 淮北 / 银川）对比作为 Chapter 6 候选，状态 OPTIONAL。

### 13.5 一句话叙事

> 本研究在 EnergyPlus + Sinergym 数据中心仿真平台上构建了带 TES + PV + ToU 信号的联合环境，先用 supervisory MPC-lite 证明 TES 峰谷 mechanism 在固定非 TES action 下物理可达（trainlike 672-step：charge_window_sign_rate 1.0、Δ_SOC_prepeak +0.25），再以系统性的纯 RL 失败复盘说明短视 SAC/DSAC-T 在 delayed inventory control 上的 credit assignment 局限，并把 MPC oracle 的输出作为 TES-head BC/IL warm-start 的标签源，为后续无 runtime planner 的 learned policy 提供可解释的上界基线。

## 14. 2026-05-02 M2-F1 主线切换：TES MPC-lite
