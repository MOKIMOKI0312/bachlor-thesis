# 复现问题根因分析

## 核心发现

**作者发布的代码和作者实际训练时使用的代码不一致。** 我们用作者发布的代码训练，和作者的预训练模型对比时，实际上是在比较两个**不同问题**的解。

---

## 证据链

### 证据 1：作者的预训练模型有活跃的电池循环，我们的没有

| 观测维度 | 变量 | 作者 Singapore 模型 mean | 我们的模型 mean |
|---------|------|----------------------|---------------|
| dim19 | SoC（电池荷电状态） | **202,121** | **2,208**（初始值，恒定） |
| dim20 | act_Charge（充电功率） | **10,536,000** W | **0** |
| dim21 | act_Discharge（放电功率） | **9,913,106** W | **0** |
| dim24 | ElectricityProduced | **21,098,662,920** J | **3.006** J |

我们的训练数据中电池从未充放电（SoC 始终为初始值 2208），发电量为 3 焦耳（≈0W）。
作者的模型训练时电池活跃循环，发电量均值约 21GJ/步 ≈ 5.8MW。

### 证据 2：发电机在 epJSON 中是占位符

GitHub 仓库和我们本地的 `DRL_DC_training.epJSON` 中：
```json
"Generator:PVWatts": {
    "PV_farm": {
        "dc_system_capacity": 0.001    // ← 0.001 瓦！
    }
}
"Generator:WindTurbine": {
    "WT1": { "rated_power": 0.001 }    // ← 0.001 瓦！
}
```

0.001W 的 PV 产生不了任何有意义的电力，电池也就没有能量来源去充电。

### 证据 3：电池通过 EMS 被显式禁用

epJSON 中的 EMS Program P_3 逻辑：
```
IF Charge_Def == 0
  SET Discharge_rate = 0
  SET Charge_rate = 0
```

`Charge_Def_Set` 的默认值 = 0，且在 `Eplus-DC-Cooling` 环境注册中**没有**为 `Charge_Def_Set` 注册 actuator（只有 `Eplus-DC-Grid` 才有 `Charge_Def_DRL`）。所以电池永远被禁用。

### 证据 4：作者的 ElectricityProduced 随地点变化，符合真实 PV

| 站点 | 作者 ElectricityProduced mean | 太阳辐射强度 |
|------|------------------------------|-------------|
| Dallas | 2.196 × 10^10 | 高（美国南部） |
| Singapore | 2.110 × 10^10 | 高（赤道） |
| Sydney | 1.894 × 10^10 | 中高 |
| Frankfurt | 1.879 × 10^10 | 中 |
| Lulea | 1.816 × 10^10 | 低（瑞典北部） |

发电量和站点太阳辐射强度的地理排序一致。如果是 0.001W 的占位符 PV，所有站点应该几乎为零且无差异。这证明**作者训练时使用了真实容量的 PV 系统（约 6MW）**。

### 证据 5：文件大小不一致

| 文件 | GitHub 原始 | 我们本地 |
|------|-----------|---------|
| DRL_DC_training.epJSON | 282,223 bytes | 288,052 bytes |

我们的文件比 GitHub 上的多了约 5.8KB。可能是 migration 过程中被修改了，但关键的 PV 容量在两者中都是 0.001W。

### 证据 6：作者有 10 个不同的预训练模型


## 因果链：为什么我们的"通用模型"表现更好

```
作者实际训练 → 真实 PV(~6MW) + 活跃电池 + 单站点天气 → 模型学会管理 PV/电池/冷却
作者发布代码 → 占位 PV(0.001W) + 电池禁用
我们用发布代码训练 → 无 PV/电池 → 模型只学冷却控制
评估用发布代码的 epJSON → PV=0.001W, 电池死亡

评估时：
  作者模型：看到 SoC=2208(恒定), Charge=0, Discharge=0
            归一化后 SoC=(2208-202121)/√4.7e9 = -2.92σ（极端异常值）
            每个时步都收到"电池极度亏电"的信号 → 策略输出有系统偏差

  我们模型：看到 SoC=2208(恒定), Charge=0, Discharge=0
            归一化后 SoC=(2208-2208)/√0.0002 ≈ 0（正常值）
            这些维度对策略没有干扰 → 纯粹的冷却优化
```

**结论**：不是我们更强，而是作者的模型在错误的评估环境中受到了系统性惩罚——它的 3 个观测维度收到了训练时从未见过的异常值。

## 对你的毕设的影响

### 这次对比不可用

你**不能**在论文中写"我们的模型优于作者的模型"，因为：
1. 两个模型训练的是不同的问题（有/无 PV+电池）
2. 评估环境与作者的训练环境不匹配
3. 作者模型在评估中受到了不公平的归一化偏差惩罚

### 你的训练结果仍然有效

你的模型在**无 PV、无电池、占位符发电机**的环境中是一个有效的冷却控制器。它可以作为你后续扩展（TES + IT 调度 + 真实 PV）的基线。

### 正确的复现路径

由于作者没有公开真实的训练 epJSON（PV 容量、电池参数等），**严格复现作者的绝对结果是不可能的**。但你可以：

1. **接受这个差异**，在论文中说明：
   > "作者发布的代码中 PV/Wind 容量为占位符值（0.001W），与预训练模型的训练环境不一致。本文基于发布代码重新训练了冷却控制基线。"

2. **以你自己的训练结果为基线**，后续所有扩展实验（E0-E6）都在同一个环境中做对比，确保实验内部一致性。

3. **对于 Singapore 站点**，用单站点训练替代通用训练（见下方），让基线更清晰。

### 立即要做
