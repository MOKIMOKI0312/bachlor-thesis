#### 作者原始代码

作者的 `DRL_training.py`（我们直接使用的同一文件）中使用了完全相同的 10 个天气文件列表。作者仓库 README 中没有提到额外的天气文件配置。`__init__.py` 中的默认注册只用了 Sydney 单站点（`AUS_NSW_Sydney...epw`），但那是环境注册默认值，训练脚本会覆盖它。

#### 模型表现差异的可能原因

既然天气配置相同，我们的 seed10 优于作者模型的可能原因：
1. **随机种子差异**：我们跑了 12 个 seed，取了最好的；作者可能只跑了少量 seed
2. **训练轮数差异**：作者可能用了不同的 episode 数或提前停止策略
3. **评估中的 workload trace 越界**（见问题 2）：PAI2020 trace 有值 >1.0，可能影响了不同模型的表现差异
4. **checkpoint 选择策略**：我们用了探针系统选 best checkpoint，作者可能用了最终模型

**建议**：这个差异不影响复现的有效性。后续扩展实验中，我们只需要和自己的基准（seed10 在 Singapore 单站点）做对比。

## 问题 2：评估脚本中的 workload trace 越界 bug

### 结论

**PAI2020 和 Venus 两个 trace 存在 >1.0 的值，会通过 `action[5]` 直接注入 EnergyPlus actuator，超出 action space 上界。需要加 clip。**

### 调研详情

#### 各 trace 的值域

| Trace | 最大值 | >1.0 的数据点数 | 用于评估? |
|-------|--------|----------------|----------|
| Earth | 0.767 | 0 | 是 (Sydney) |
| Kalos | 0.383 | 0 | 是 (Singapore) |
| PAI2020 | **1.226** | **20** | 是 (Frankfurt) |
| Saturn | 0.789 | 0 | 是 (Lulea) |
| Seren | 0.711 | 0 | 否 |
| Uranus | 0.761 | 0 | 否 |
| Venus | **1.199** | **2** | 否 |

#### bug 位置

`compare_author_vs_checkpoint.py:88` 和 `DRL_evaluate.py:94`：
```python
action[5] = util_rate[steps]  # 直接写入，无 clip
```

`sinergym/envs/eplus_env.py:398-405` 中只做了 warning，不做 clip：
```python
try:
    assert self._action_space.contains(action)
except AssertionError as err:
    self.logger.warning('Step: The action {} is not correct...')
```

#### 影响

- action space 定义为 `Box(low=[0]*7, high=[1]*7)`，值 >1.0 超出上界
- 该值通过 `simulators/eplus.py:327-330` 直接写入 EnergyPlus actuator（`ITE_Set` Schedule），EnergyPlus 会对超出 Schedule 范围的值报 warning
- 对 Frankfurt:PAI2020 case（20 个越界点）影响最大，可能导致该 case 的 reward 偏低

#### 修复方案

在 `compare_author_vs_checkpoint.py:88` 和 `DRL_evaluate.py:94` 处加 clip：

```python
# 修复前
action[5] = util_rate[steps]

# 修复后
action[5] = np.clip(util_rate[steps], 0.0, 1.0)
```

#### 影响评估

修复后 Frankfurt:PAI2020 case 的 reward 预计会变化（20 个数据点被 clip 到 1.0），其他 3 个 case 不受影响。需要重新跑 4-case 评估生成修正后的对比结果。

## 问题 5：Sinergym 的 EMS Actuator 注册机制

### 结论

**系统采用两层间接架构：Sinergym 将 SAC [0,1] 原始值写入 Schedule:Constant "信箱"，EnergyPlus 内部的 EMS Program 读取信箱并做滑动窗口缩放，转换为物理设定值。新增 actuator 需要四步：epJSON 加 Schedule + EMS Sensor + EMS Actuator + EMS Program 缩放逻辑，再在 `__init__.py` 注册。**

### 重要发现

1. **Chiller_Pump（action[1]）实际上是无效的**——EMS Sensor 存在但没有任何 EMS Program 使用它，泵按默认/autosize 行为运行
2. **ITE（action[5]）在训练时被忽略**——EMS Program P_2 在训练模式下使用随机游走生成 IT 负荷，仅在评估时直接使用 SAC 输出

### 完整 action 传递链路

```
SAC 输出 action[0..6] (float32, [0,1])
  │
  ▼ Sinergym 侧（无缩放）
  env.step(action)
    → self.act_queue.put(action)                    # eplus_env.py:439
      → _process_action() callback                  # simulators/eplus.py:298
        → exchange.set_actuator_value(handle, action[i])  # 写入 Schedule:Constant
  │
  ▼ EnergyPlus 侧（EMS Program 做滑动窗口缩放）
  EMS:Sensor 读取 Schedule:Constant 值
    → EMS:Program 应用 sliding window 缩放
      → EMS:Actuator 写入真实 HVAC 设定点
```

### EMS Program 缩放逻辑（关键！）

EMS Program `P_1` 在 `BeginTimestepBeforePredictor` 时执行，对 HVAC 控制量应用**滑动窗口增量控制**：

```
physical_value = (dynamic_max - dynamic_min) × action[0,1] + dynamic_min

其中:
  dynamic_min = max(当前值 - Δ, 绝对下界)
  dynamic_max = min(当前值 + Δ, 绝对上界)
```

各控制量的参数：

| action 索引 | 控制量 | 绝对范围 | 每步最大变化 Δ | 安全超控条件 |
|------------|--------|---------|---------------|------------|
| 0 | CRAH 风机流量 | 500-2000 kg/s | ±100 kg/s | 室温>30°C → 2000 |
| 1 | **冷冻水泵** | **无效！** | — | — |
| 2 | 冷凝水泵流量 | 300-5000 kg/s | ±100 kg/s | 室温>30°C → 5000 |
| 3 | CRAH 送风温度 | 11-22°C | ±0.5°C | 室温>30°C → 10°C |
| 4 | 冷机出水温度 | 5.6-10°C | ±0.5°C | 室温>30°C → 5.6°C |
| 5 | IT 利用率 | [0,1] | 训练：随机游走；评估：直接使用 | — |
| 6 | 电池充放电 | 基于 SoC 和电压动态计算 | — | Charge_Def flag 门控 |

**安全超控**：当机房温度 > 30°C 时，P_1 忽略 SAC 输出，强制将所有 HVAC 设定到最大冷却状态。

### 当前 7 个 actuator 的注册方式

`sinergym/__init__.py` 中 `Eplus-DC-Cooling` 的注册：

## 问题 6：SB3 训练指标输出不完整

### 结论

**`actor_loss` / `critic_loss` 为空是正常的——SB3 的 SAC 在 `learning_starts`（8760 步，即第 1 个 episode）之前不做梯度更新，自然没有 loss。之后 loss 的输出频率由 `log_interval` 控制。**

### 调研详情

#### 当前配置

`DRL_training.py:142` 和 `run_short_training.py:123`：
```python
model.learn(
    total_timesteps=total_timesteps,
    log_interval=500,   # 每 500 次 gradient update 输出一次到 SB3 logger
    callback=CallbackList(callbacks),
)
```

SAC 的超参数：
```python
model = SAC(
    'MlpPolicy', env,
    learning_starts=8760,  # 前 8760 步（1 个 episode）只收集数据不训练
    ...
)
```

#### `log_interval` 的含义

SB3 中 `log_interval` 的单位是 **rollout 次数**（对 off-policy 算法如 SAC，即 episode 数），不是 gradient step 数。`log_interval=500` 意味着**每 500 个 episode 输出一次训练指标到 SB3 logger**。

对于 300 episode 的训练，`log_interval=500` 意味着**从不输出**。

#### 修复方案

将 `log_interval` 改为更小的值：

```python
model.learn(
    total_timesteps=total_timesteps,
    log_interval=1,    # 每个 episode 结束时输出一次训练指标
    callback=CallbackList(callbacks),
)
```

#### 探针系统的匹配

检查 `training_monitor.py` 中是否从 SB3 的 logger 读取 loss，还是从 callback 中读取。如果探针依赖 SB3 的 tensorboard/csv logger，那么 `log_interval=1` 足够。如果探针通过 callback 的 `self.model.logger` 读取，则需要确认 callback 的调用频率。

#### 影响

这个问题**不影响训练质量**，只影响监控的可视化。修复后下次训练即可看到完整的 loss 曲线。当前训练结果不需要重新跑。
