## 4. 关键发现与限制（M2 必须吸收）

### 4.1 TES 未激活 — 根因分析

奖励函数（`sinergym/utils/rewards.py` 的 `PUE_TES_Reward`）只看**瞬时**功率 + comfort 惩罚 + SOC 越界软惩罚：

```
reward = - λ_energy · P_facility(t) / P_IT(t)  # PUE 项
         - λ_temp · max(0, T - 25)^0.5          # comfort sqrt 惩罚
         - λ_soc · (SOC 越界时的软惩罚)          # SOC 软边界
```

问题：
- **没有时段差异**：P(t) 无论在何时都等权，TES 在凌晨充冷和正午放冷对 reward 贡献相同
- **没有电价乘子**：load shifting 的真实经济价值完全不进入 reward
- **没有 PV 信号**：更无法做"光伏高峰时优先用 PV 电"的决策

观测空间（22 维）中**没有时间信号**（hour_of_day / month）：
- policy 输入无法区分"凌晨 3 点"和"下午 3 点"
- 即使 reward 有电价，policy 也不知道现在电价高低

**结果**：DSAC-T 学到的最优解是"让 TES 维持一个稳态 SOC（~0.29），valve 常数微开以防止 SOC 自然漂移"。这是一个数学上合理但物理上无用的 policy。

### 4.2 E0.3 比 E1-R2c 更好 — 反常现象

PUE 1.3071 vs 1.3505，E0.3 领先 3.32%。可能的原因：

1. **多一维动作的负担**：E1-R2c 6 维动作 vs E0.3 5 维，更大的动作空间在相同训练预算下探索不足
2. **TES 引入的拖累**：无电价激励时，policy 需要花容量去忽略 TES 维度，HVAC 部分训练不充分
3. **comfort 权重不对称**：E0.3 用 `λ_temp=1.0`，E1-R2c 用 `λ_temp=3.0`（为应对 TES 过放电过热风险），导致 E1-R2c agent 更保守，PUE 优化不激进

**论文建议**：把 E0.3 作为 main baseline（12 seed × 300 ep，PUE 1.3071 稳定），E1-R2c 作为 "M1 ablation"（加 TES 无电价的反面教材），E3/E4 作为 main result。

### 4.3 E0.3 model.zip 已删

磁盘清理时没保护 E0.3 的训练 checkpoint。**后果**：
- 无法用 E0.3 warm start（需要 model 权重）
- 无法对 E0.3 model 做新的评估（比如新环境、新气候）
- **但 JSON 评估结果 + 每小时时序 CSV + wandb 曲线都在**，论文引用够用

若 M2 想做 E0.3 vs E3 的**物理指标对比**（PUE/comfort/电费），可以直接用 `results/e03_evaluation_summary.json` 做参考数字，不需要重跑 E0.3。

## 8. 已知陷阱（从 M1 学到的教训，M2 别重复）

### 8.1 SB3 `reset_num_timesteps=False` 语义

**坑**：脚本作者以为 `total_timesteps` 是 "已训 + 增量 = 累计目标"，于是计算 `total = already_done + new_episodes * steps`。但 SB3 内部在 `_setup_learn()` 里还会再 add `self.num_timesteps` 到这个 total 上，导致实际 target = `2 × already_done + new_episodes × steps`。

**结果**：seed8 本应 target ep 305，实际训到 ep 314-319。

**修复**：`tools/run_tes_training.py:132` 改为：
```python
timesteps = episodes * timesteps_per_episode  # 不加 model.num_timesteps
```

### 8.2 E+ loop pump 和 branch pump 不共存

**坑**：EnergyPlus 23.1 限制一个 PlantLoop 里 loop-level Pump（如 `Chilled Water Loop Secondary Pump`）和 branch-level Pump 不能共存。给 TES 加专用 `Pump:VariableSpeed` 会 fatal error。

**当前 workaround**：用 EMS `System Node Setpoint` actuator 直接控制 TES use/source 侧节点的 mass flow rate max/min，由 EMS Program P_5 根据 agent 的 `TES_Set` 信号计算目标流量（Max flow = 97.2 kg/s = 1400 m³ / 4h）。**此方案 M2 继续沿用，不用改**。

### 8.3 monitor.csv 列重名

**坑**：LoggerWrapper 构造 header 时，如果 wrapper 链里多次追加相同字段名（如 TESIncrementalWrapper 和底层环境都写 `TES_valve_position`），最终 header 有两列同名。`csv.DictReader` 遇重名取最后一列的值。

**绕过**：评估脚本用 `pandas.read_csv(header=None)` + 位置索引，或在 LoggerWrapper 构造时 dedupe header。

### 8.4 Git Bash `nohup ... &` 在 Windows 不 detach

**坑**：Git Bash for Windows 的 `nohup cmd > log.txt &` 虽然 fork 出 shell job，但父 bash 退出时子进程也会被 kill（不是真 detach）。

**修复**：用 `tools/launch_eval_background.py` 里的 Python Popen + `CREATE_NEW_PROCESS_GROUP | DETACHED_PROCESS` 标志，或 `tools/launch_training_background.py` 的相同方案。

### 8.5 `ACE-CORE102700` filter driver 不稳定

**坑**：2026-04-17 02:30 系统卡死前出现 `Microsoft-Windows-FilterManager ID=11` 警告：`文件系统筛选器 ACE-CORE102700 不支持绕过 IO`。这是某反作弊 / 云盾软件的 minifilter driver。在 8 seed × E+ 并发 IO 压力下疑似触发 DPC 超时，导致系统缓慢 shutdown。

**缓解**：
1. M2 训练并发 ≤ 4 seed
2. 可查找 `ACE-CORE` 是哪个软件（可能是 Alibaba Cloud Enterprise / 某游戏反作弊），训练期关掉服务
3. 用 BlueScreenView（NirSoft）分析历史 minidump 找确切元凶

### 8.6 Windows `date` 命令时区可能错乱

**坑**：Git Bash 里 `date` 命令输出的时间可能与 PowerShell `Get-Date` 不一致（时区配置问题）。

**绕过**：统一用 PowerShell 取时间：
```bash
powershell -Command "Get-Date -Format 'yyyy-MM-dd HH:mm:ss zzz'"
```

### 8.7 挂起训练可能加剧系统不稳定

**坑**：本次会话 04-18 用户挂起训练（NtSuspendProcess）后约 17 分钟，系统 Event 41 意外重启。怀疑 6 GB 挂起 python 进程 + `ACE-CORE102700` 监控 + 训练临时文件锁定共同作用。

**建议**：M2 时若要让出电脑，**优先用"kill + 从最近 ckpt resume"**（本次新建 `launch_training_background.py --resume` 支持），而不是挂起。

### 8.8 epJSON / 环境改动必须走 `eplus-modeler` subagent

参考 `memory/subagent_protocol.md`（如果用户的 memory 目录里有）。任何改 `Data/buildings/*.epJSON` 的工作都要让 eplus-modeler agent 执行，同步更新 `Data/buildings/建筑模型说明.md`。
