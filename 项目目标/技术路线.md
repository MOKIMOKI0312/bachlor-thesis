# 毕业设计技术路线文档

## 项目概述

项目名称：面向光伏自消纳的 AI 数据中心冷却-蓄冷-算力联合优化系统设计。

核心研究问题：在 EnergyPlus + Sinergym 的数据中心仿真平台上，证明蓄冷水罐（TES）峰谷套利机制在固定非 TES action / fixed fan / trainlike `ITE_Set=0.45` 条件下可达，并对比"显式 anticipatory 控制（MPC）"与"末端 reward 驱动的 RL"在该 mechanism 上的可达性差异。

技术栈：EnergyPlus 23.1 + Sinergym（Xiao & You 修改版）+ `scipy-highs` LP 求解器 + Stable-Baselines3（DSAC-T，仅对照） + Gymnasium。

站点：**中国江苏南京**（见 §2.5 电价 / §2.4 PV / §2.6 气象）。

## §1 EnergyPlus 物理层（共享基础，不随主线切换变动）

Xiao & You 未提供 IDF 文件，仅提供 epJSON 文件。epJSON 是 JSON 格式，结构化程度高于传统 IDF，可直接用文本编辑器和 Python 脚本操作，不需要 OpenStudio。

### 1.1 水罐容量

参考"其它参考文献"中 Zhu 的论文，建议对于 12.3 MW 的数据中心设置 1400 m³ 储水罐，因此本项目以 1400 m³ 为基准。M1 阶段曾尝试 4300 m³ 大容量（基于"4h × 峰值冷却"工业指南），M1-R2 后回到 1400 m³ 与 EnergyPlus EMS workaround 适配。

### 1.2 删除电池模块

在 epJSON 中搜索以下关键词，找到所有相关对象并整体删除：

- `ElectricLoadCenter`
- `Generator`
- `ElectricalStorage` / `Battery`

删除时检查是否有其他对象引用了被删对象的节点名或 schedule 名，一并清理。由于 PV 出力改为外部预计算注入（见 §2.4），整个 `ElectricLoadCenter` 可移除。删完后跑一次 EnergyPlus 验证无 orphan node 错误。

### 1.3 增加储冷罐 Branch

在 CHW Plant Loop 供水侧新增一条并联 branch，挂载 `ThermalStorage:ChilledWater:Stratified`。需要新增的 epJSON 对象：

- `ThermalStorage:ChilledWater:Stratified`（水罐本体，1400 m³，分层节点 6–10 个，初始温度 10°C）
- `Branch`（TES use/source 侧）
- 相关节点（inlet/outlet node）
- 供水侧 `BranchList`：加入新 branch 名称
- `Connector:Splitter` / `Connector:Mixer`：加入新 branch 的进出节点
- `PlantEquipmentOperationSchemes`：将 TES 加入调度序列

**实施注记（2026-04-17）**：EnergyPlus 23.1 限制一个 PlantLoop 内 loop-level pump 和 branch-level pump 不能共存。本项目 CHW Loop 已有 loop pump（Chilled Water Loop Secondary Pump），无法再添加独立 TES 变速泵。替代方案：使用 EMS `System Node Setpoint` actuator 直接控制 TES use/source 侧节点的 mass flow rate max/min，由 EMS Program P_5 根据外部 `TES_Set` 信号计算目标流量（Max flow = 97.2 kg/s = 1400 m³ / 4h）。此为 workaround，非 E+ 推荐做法，流量精度需场景测试（v=+0.5 / v=−0.5 / v=0）验证。

### 1.4 EMS Actuator

将 TES 流量与 SOC 控制权暴露给上层控制器（无论 MPC 还是 RL）：

- `EnergyManagementSystem:Actuator`：TES use/source 侧节点 mass flow rate
- `EnergyManagementSystem:Sensor`：罐体温度、各节点温度（用于 SOC 计算）
- `EnergyManagementSystem:Program`：P_5 接收外部信号 → 计算目标流量；P_6 计算 SOC（10 节点温度加权平均）；P_7 强制 Chiller 关停时的 HVAC 迭代内循环

同时为 IT 负荷 schedule 预设一个 EMS actuator，以便后续完整阶段的算力调度能将弹性 IT 功率回传给 EnergyPlus。

### 1.5 降级预案

若分层模型在 2 周内无法跑通 plant loop，退到 `ThermalStorage:ChilledWater:Mixed`（非分层模型），配置更简单，对 SOC 层面的调度行为研究差异很小。M1 阶段验证分层模型工作，无需启用降级。

### 1.6 验收标准

修改后的 epJSON 在 EnergyPlus 中能独立跑完一年仿真，无 fatal/severe error，TES 的 SOC 曲线在输出报告中可见且物理合理。

## §2 共享环境层（Sinergym wrapper 链与观测/动作）

以下 41 维 obs / 6 维 action 是**完整环境上界**，作为后续完整 workload 联合优化阶段的目标。M2-F1 当前阶段主线（无论 MPC 还是 RL 对照）实际暴露 32 维 obs / 4 维 action（fixed fan 折叠成常数；workload action 关闭；TES 为 action 第 4 维）。MPC 主线只控 TES 维，RL 对照在论文复盘时仍可使用全 6 维。

### 2.1 弹性 IT 算力调度（SustainDC 移植，工作上界）

整体移植 SustainDC（Sarkar et al., NeurIPS 2024）的 Workload Environment 作为 Gymnasium Wrapper。该模块负责 delay-tolerant workload 的日内时间调度，维护任务队列状态（backlog 追踪、task age histogram、日内守恒约束）。

将 SustainDC 中基于碳强度（CI）的调度信号替换为本研究的电价信号和 PV 出力信号（见 §2.4 / §2.5），保留任务队列管理机制。SustainDC 原始 `_create_ls_state()` 输出 26 维观测，按本项目改造后缩减为 24 维（CI 信号 8 → price/PV 信号 6）。

**算力任务执行结果回传 EnergyPlus**：Workload Environment 每步输出实际 IT 总功率 P_IT(t) = P_rigid(t) + P_flex(t)，通过 Sinergym action 接口写入 EnergyPlus 的 `ElectricEquipment` schedule actuator，使 IT 负荷调度的冷却代价在物理仿真中被完整反映。

**M2-F1 注意**：M2-F1 主线 fixed `ITE_Set=0.45`，不打开 workload action。Workload wrapper 仍在 wrapper 链中（保持观测维度上界），但其离散动作不被 MPC/RL 控制器选择。

### 2.2 TES 控制（动作维度与物理意义）

TES action 定义为阀门开度增量 Δv ∈ [−δ_max, +δ_max]，递推更新：

$$v(t+1) = \text{clip}\big(v(t) + \Delta v(t),\; -1,\; +1\big)$$

其中 v < 0 表示充冷，v > 0 表示放冷，v = 0 表示待机。

增量参数化的依据：

- **物理合理性**：分层水罐要求入口 Reynolds 数控制在 400–600 以内、Froude 数不超过 2（ASHRAE 设计准则）。过激的流量变化破坏 thermocline，降低蓄冷效率。
- **rate limiter**：增量上限 δ_max 起到流量变化速率限制的作用。
- **MPC 中的处理**：MPC 求解器在 LP 中显式给出每步 Δv 上下界（受 rate limit + SOC guard 共同约束），首步执行后递推；不再依赖 RL 学习增量先验。

δ_max 取 0.15–0.25。`TESTargetValveWrapper` 是 wrapper-level rate limit 与 SOC guard 的最终 actuator 可行性层，**MPC 与 RL 都通过它落到 EnergyPlus**。

### 2.3 PV 信号

#### 2.3.1 设计原则

PV 出力作为外生时间序列注入观测空间，控制器（MPC 或 RL）无需学习气象→PV 的物理映射。理由：

1. 论文核心是调度方向，PV 是外部驱动信号；
2. PVGIS 已用确定性 6 参数效率模型精确描述气象→PV 映射；
3. RL 领域将不可控外部信号视为 exogenous data 是标准做法（Ju & Crozier 2024）。

#### 2.3.2 数据来源

使用 PVGIS 5.2（pvlib `get_pvgis_hourly()`），Nanjing 坐标（32.0584°N, 118.7965°E），6 MWp 装机，27° 固定倾角（南向），c-Si 组件。年发电 7.14 GWh（1191 kWh/kWp）。生成工具 `tools/generate_pvgis.py --site nanjing --target-year 2025`，输出 `Data/pv/CHN_Nanjing_PV_6MWp_hourly.csv`。

## §3 TES MPC-lite 主线

主线由两个工具组成：

| 工具 | 角色 | 输出 |
|---|---|---|
| `tools/m2_tes_mpc_oracle.py` | M1 oracle，证明 trainlike 物理可达 | `runs/m2_tes_mpc_oracle/<tag>/{monitor.csv, result.json}`、`analysis/m2f1_mpc_oracle_summary_202605.{csv,json}` |
| `tools/evaluate_m2_mpc_lite.py` | M2 supervisory 可报告控制器 | `runs/m2_mpc_lite/<tag>/{monitor.csv, result.json}` |

参数与 CLI 来源：`analysis/m2f1_mpc_lite_technical_route_202605.md`。

### 3.1 控制问题表述

- **决策周期**：15 min（与 EnergyPlus 4 timesteps/hour 对齐）。
- **预测/规划水平**：24 h rolling horizon（96 step），首步执行（receding horizon）。
- **状态**：TES SOC（连续 [0,1]）、当前阀门 v、当前/未来 TOU 价格窗口、当前/未来 PV ratio。
- **决策变量**：每步 Δv（阀门增量），等价隐含的 charge/hold/discharge 模式标签（写入 monitor 用于审计）。
- **输入预测**：TOU 与 PV 是确定性年时间序列，rolling window 直接读未来 96 step；不引入 stochastic forecast 不确定性建模。

### 3.2 优化目标

$$
J = \sum_{k=1}^{N} \big[\, c_{\text{TOU}}(k) \cdot p_{\text{net}}(k) + \lambda_{\text{term}} \cdot |\text{SOC}_N - \text{SOC}_{\text{target}}| + \lambda_{\text{sched}} \cdot |\text{SOC}_k - \text{SOC}_{\text{schedule}}(k)| + \lambda_{\text{slack}} \cdot s(k) \,\big]
$$

- 第一项：rolling cost arbitrage（TOU × 净购电）。
- 第二项：terminal SOC penalty（防止 oracle 把所有库存放空换短期收益）。
- 第三项：SOC schedule penalty（高价段目标 SOC 低、低价 pre-peak 段目标 SOC 高，引导日内库存形态）。
- 第四项：高惩罚 SOC slack（保证 LP 在 SOC 触界时仍可行，slack 量被记录而非隐藏）。
- **comfort 项**：当前为零权重 placeholder。M2-F1 不在 LP 中建模 comfort 风险；wrapper guard 是最终安全层。后续若要建模需引入可追溯的热风险模型（method_takeaways §"Constraints Added After Review"）。

### 3.3 约束

- SOC 物理界：[0.05, 0.95]（保守 wrapper bounds 是 [0.10, 0.90]，差值是 LP 安全裕度）。
- Rate limit：|Δv| ≤ δ_max（δ_max = 0.20）。
- 模式约束：
  - low-price 段：TES discharge 上界 = 0（不允许在低价时放冷套利失败）。
  - low-price pre-peak 且 SOC 有 headroom：当前 step charge 下界（强制 anticipatory 充冷）。
  - high-price 段：TES charge 上界 = 0。
  - high-price 且 SOC 有库存：当前 step discharge 下界（强制释放）。
- SOC schedule：高价段目标低、未来 pre-peak 之前 headroom 目标、low-price pre-peak 中目标高。
- `safety_recovery` 标签：当 EnergyPlus 实际 SOC 已落在严格 wrapper bounds 之外，恢复动作单独标记，不与正常的 low-price discharge 混在一个统计桶里。

### 3.4 求解器

> **当前 master HEAD (`2b0a3b78`) 现实（2026-05-04 注）**：`tools/m2_tes_mpc_oracle.py` 的 `parse_args()` line 1550–1552 后置硬编码 `args.solver = "heuristic"`，三连覆盖 `solver / solver_used / solver_status`。`plan_tes_action_scipy_highs(...)`（LP 实现，line 445）写在那但 CLI 不可达。所以 §3.7 表中所有"PASS"实际是 heuristic 路径跑出来的。**目标设计**（下面这段）是 W1-0 解锁 LP CLI + W1-2 加 MILP 后 master 的状态。

**目标设计**：

- **首选（W1-2 上环后）**：`scipy.optimize.milp`（HiGHS MILP，开源、deterministic，binary charge/discharge 模式互斥消除连续松弛伪解）。每个 rolling step 运行一次 24 h MILP，求解状态写入 `solver_status_counts`。
- **次选（W1-0 解锁后）**：`scipy-highs` LP（HiGHS linprog，无 binary）。MILP 不可用时使用，作为 LP-relaxation baseline。
- **fallback**：deterministic rolling heuristic（基于 charge/hold/discharge 模式判断 + 终端 SOC 修正）。当 LP/MILP 报 infeasible 或不可用时切到此层，并在 `result.json["solver_status_counts"]["milp_to_heuristic_fallback"]` / `lp_to_heuristic_fallback` 显式记录计数。
- **输入校准**：SOC 动力学增益从已有 oracle monitor 标定，或由 CLI `--soc-gain` 显式提供；fallback 显式记录于 result。
- **CLI**：`--solver {heuristic, lp_highs, milp}`，默认 `milp`（W1-0 之前默认行为为 heuristic，W1-0 之后切到 milp）。

### 3.5 与现有 wrapper 的关系

MPC 控制器在 wrapper 链外侧产生 Δv 目标，仍**通过 `TESTargetValveWrapper` 注入 EnergyPlus**。该 wrapper 提供：

- valve rate limit（即使 MPC 给出违法 Δv，wrapper 会 clip）；
- SOC guard（即使 MPC SOC 预测偏离，wrapper 会强制 actuator 安全）；
- monitor 列：MPC 的 `intended_target` 与实际 `actuator_value` 分别记录，差异即为 wrapper 干预量。

这一层的存在是文献 ICCPS22 安全 DRL 与 Jeong 2024 ESS arbitrage clipping 一致性的工程要求：**clipping 必须可见、可审计**，不允许 MPC 与 wrapper guard 之间出现"静默修正"。

### 3.6 mechanism gate（M2-F1 trainlike 672-step）

| 指标 | gate |
|---|---|
| `comfort` | < 3 % |
| `guard` | < 5 % |
| `charge_window_sign_rate` | ≥ 0.80（feasible low-price pre-peak 窗口） |
| `low_price_discharge_fraction` | ≤ 0.10 |
| `charge_window_valve_mean` | < −0.05 |
| `discharge_window_valve_mean` | > +0.05 |
| `delta_soc_prepeak` | > 0 |
| `delta_soc_peak` | < 0 |

`charge_window_sign_rate` 与 `delta_soc_prepeak` 限定到 feasible low-price pre-peak 窗口（episode 起始 SOC 已满时不算合法 charge opportunity）。Infeasible 行保留在 monitor 并标 reason，不被算入分母。

## §4 RL 对照路线（附录 A，从原 RL 主线降级保留）

RL 路线在 M2-F1 阶段经过 C1/C2/C2b/C3/A-prime/O1/P1a BC warm-start 等多轮失败实验后**降级为对照/辅助**。本节保留其设计描述与失败证据，供：

(i) 论文 Chapter "失败分支复盘"叙事使用；
(ii) 后续若做 MPC-label TES-head BC/IL warm-start 时作为对照基线；
(iii) 完整环境上界（41 obs / 6 action）的语义文档参照。

### 4.1 RL-Cost reward

$$r_{\text{cost}}(t) = -\alpha \cdot P(t) \cdot \text{LMP}(t) - \beta \cdot \text{comfort\_penalty}(t) - \gamma \cdot \text{soc\_penalty}(t)$$

其中 P(t) 来自 obs 直读 `_energy_MWh(obs_dict)`，不通过 `cost / price` 反推（避免负价时 mwh = 0 静默退化，commit `a5898e7` 修复）。

### 4.2 RL-Green reward

$$r_{\text{green}}(t) = -\alpha \cdot P(t) \cdot \min\big(\text{LMP}(t),\, c_{\text{pv}}(t)\big) - \beta \cdot \text{comfort} - \gamma \cdot \text{soc}$$

PV 出力 ≥ 阈值时 `effective_price = min(LMP, c_pv)`，否则退化为 LMP。虚拟绿电价格 c_pv ≈ 0 引导 agent 把负荷移向太阳能高峰时段。**虚拟价格只在 reward 中出现，不进入观测**——agent 看到的是真实 LMP 和 PV 出力。

### 4.3 SAC / DSAC-T 选择

E0.2 阶段标准 SAC 12 seed 大面积发散（仅 2/12 健康），文献调研（Ali 2021、SSAC MDPI Buildings 2025）确认 SAC 自动熵调节是 HVAC RL 已知不稳定来源。E0.3 切到 **DSAC-T**（Distributional SAC with Three Refinements，arXiv:2310.05858），R3 方差自适应梯度缩放显著抑制 ent_coef 飙升，12/12 健康。E0.3 PUE mean 1.3071，相对 baseline 1.3973 改善 6.45 %（无 TES 无电价）。

### 4.4 M2-F1 失败证据

主要失败实验（详见 `毕业设计项目进度/代码开发进度管理.md` §7.7）：

- **P1a `direction_amp_hold` + `rule_tou` BC warm-start**（34 obs / 5 action）：672-step `charge_window_sign_rate ≈ 0.103`、`low_price_discharge_fraction ≈ 0.655`、`low_price_valve_mean ≈ +0.323`、`high_price_valve_mean ≈ +0.376`。

## §5 论文叙事

### 5.1 主贡献

构建可复现的 EnergyPlus + Sinergym 数据中心 TES 场景（Nanjing + 江苏 TOU + Nanjing PV + 1400 m³ 分层水罐 + EMS workaround + `TESTargetValveWrapper`），并给出**可审计的 supervisory MPC-lite 控制器**，证明在固定非 TES action / fixed fan / trainlike `ITE_Set=0.45` 条件下 TES 峰谷 mechanism gate 可达（comfort 0.89%、charge_sign 1.0、Δ_SOC_prepeak +0.25、Δ_SOC_peak −0.18、PUE 1.26）。

### 5.3 可选贡献

MPC oracle monitor 生成 TES-head BC/IL warm-start labels，在 RL 训练阶段提供监督信号；**最终 learned policy 评估时不调用 runtime planner**，且必须证明 BC/runtime 权重为 0。该路线对应 §3 的下游里程碑（详见进度文档 §4 M2-G1~G3）。

### 5.4 多站点 case study（OPTIONAL）

时间余量充裕时，展示方法从市场化电价（CAISO 历史失败）到 ToU 机制（江苏 / 安徽 / 宁夏）的迁移能力。Chapter 5 可包含 CAISO 失败的"算法边界"案例，作为 DSAC-T 在批发市场重尾 reward 下稳定性挑战的开放性问题报告。
