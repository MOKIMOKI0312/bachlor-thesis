# 毕业设计技术路线文档

## 项目概述

本项目研究 reward shaping 如何影响数据中心中 TES（蓄冷水罐）和弹性 IT 负荷的调度方向，核心对比"成本驱动"（RL-Cost）与"绿电驱动"（RL-Green）两种 reward 下 agent 的行为差异。技术栈为 EnergyPlus + Sinergym + Stable-Baselines3（SAC），站点选择 Singapore。

---

## 1. EnergyPlus 层：修改 epJSON 文件

Xiao & You 未提供 IDF 文件，仅提供 epJSON 文件。epJSON 是 JSON 格式，结构化程度高于传统 IDF，可直接用文本编辑器和 Python 脚本操作，不需要 OpenStudio。


### 1.0 水罐容量确认
参考"其它参考文献"中Zhu的论文,其建议对于12.3MW的数据中心设置1400m3的储水罐,因此本项目中将1400为储水罐容量基准,并在后续实验中上下浮动50%
### 1.1 删除电池模块

在 epJSON 中搜索以下关键词，找到所有相关对象并整体删除：

- `ElectricLoadCenter`
- `Generator`
- `ElectricalStorage` / `Battery`

删除时检查是否有其他对象引用了被删对象的节点名或 schedule 名，一并清理。由于 PV 出力改为外部预计算注入（见第 4 节），整个 `ElectricLoadCenter` 可移除。删完后跑一次 EnergyPlus 验证无 orphan node 错误。

### 1.2 增加储冷罐 Branch

在 CHW Plant Loop 供水侧新增一条并联 branch，挂载 `ThermalStorage:ChilledWater:Stratified`。需要新增的 epJSON 对象：

- `ThermalStorage:ChilledWater:Stratified`（水罐本体，800 m³，分层节点 6–10 个，初始温度 10°C）
- `Pump:VariableSpeed`（TES 专用泵）
- `Branch`（将泵和水罐串联）
- 相关节点（inlet/outlet node）

需要修改的现有对象：

- 供水侧 `BranchList`：加入新 branch 名称
- `Connector:Splitter` 和 `Connector:Mixer`：加入新 branch 的进出节点
- `PlantEquipmentOperationSchemes`：将 TES 加入调度序列

**实施注记（2026-04-17）**：EnergyPlus 23.1 限制一个 PlantLoop 内 loop-level pump 和 branch-level pump 不能共存。本项目 CHW Loop 已有 loop pump（Chilled Water Loop Secondary Pump），无法再添加独立 TES 变速泵。替代方案：使用 EMS `System Node Setpoint` actuator 直接控制 TES use/source 侧节点的 mass flow rate max/min，由 EMS Program P_5 根据 agent 的 `TES_Set` 信号计算目标流量（Max flow = 97.2 kg/s = 1400 m³ / 4h）。此为 workaround，非 E+ 推荐做法，流量精度需场景测试（v=+0.5, v=-0.5, v=0）验证。

### 1.3 添加 EMS Actuator

通过 EMS 将 TES 泵流量的控制权暴露给 RL agent：

- `EnergyManagementSystem:Actuator`：获取 TES 泵流量控制权
- `EnergyManagementSystem:Sensor`：读取罐体温度、蓄冷量等状态
- `EnergyManagementSystem:Program`：接收 Sinergym/RL agent 的外部信号

同时需要为 IT 负荷 schedule 预设一个 EMS actuator，以便第 2 步的算力调度 wrapper 能将弹性 IT 功率回传给 EnergyPlus。

### 1.4 降级预案

若分层模型在 2 周内无法跑通 plant loop，退到 `ThermalStorage:ChilledWater:Mixed`（非分层模型），配置更简单，对 SOC 层面的调度行为研究差异很小。

### 1.5 验收标准

修改后的 epJSON 在 EnergyPlus 中能独立跑完一年仿真，无 fatal/severe error，TES 的 SOC 曲线在输出报告中可见且物理合理。

### 2.3 动作空间

SustainDC Workload Environment 的原始动作空间为 1 维离散动作 `spaces.Discrete(3)`：

| 动作值 | 含义 |
|--------|------|
| 0 | 延迟当前所有可延迟任务（塞入队列） |
| 1 | 不调整，正常处理当前负载 |
| 2 | 从 backlog 队列中取出任务执行 |

直接沿用此设计。离散动作与 SAC 的连续 action 空间需通过 wrapper 桥接（SAC 输出连续值后离散化，或改用 SAC-Discrete 变体）。

### 2.4 算力任务执行结果回传 EnergyPlus

Workload Environment 每步输出的实际 IT 功率 $P_{\text{IT}}(t) = P_{\text{rigid}}(t) + P_{\text{flex}}(t)$，需要回传至 EnergyPlus 以影响冷负荷计算。回传路径：

1. Workload Wrapper 根据 agent 的调度动作，计算当前 timestep 的实际 IT 总功率
2. 通过 Sinergym 的 action 接口，将 IT 功率写入 EnergyPlus 的 EMS actuator（在第 1.3 节中已预设，接管 `ElectricEquipment` 的 schedule）
3. EnergyPlus 根据更新后的 IT 功率重新计算机房热负荷和冷却系统响应

这一回传机制确保 IT 负荷调度的冷却代价在 EnergyPlus 物理仿真中被完整反映，agent 无法"免费"调度负荷。

## 3. 水罐控制建模

### 3.1 动作空间

将 TES 的 action 定义为阀门开度增量 $\Delta v \in [-\delta_{\max}, +\delta_{\max}]$，实际阀门开度通过递推更新：

$$v(t+1) = \text{clip}\big(v(t) + \Delta v(t),\; -1,\; +1\big)$$

其中 $v < 0$ 表示充冷方向，$v > 0$ 表示放冷方向，$v = 0$ 表示待机。

增量参数化的依据：

- **物理合理性**：分层水罐要求入口 Reynolds 数控制在 400–600 以内、Froude 数不超过 2（ASHRAE 设计准则）。过激的流量变化会破坏 thermocline，降低蓄冷效率。增量控制天然限制流量变化速率。
- **与 HVAC action 设计一致**：Xiao & You 的 HVAC 控制已采用增量 action（±Δ）来隐式限制设定点变化速率，TES 采用相同设计哲学。
- **替代双重平滑**：增量上限 $\delta_{\max}$ 本身起到 rate limiter 作用，不再需要一阶滤波 + 最小运行时间的叠加约束。

$\delta_{\max}$ 建议取值 0.15–0.25（从待机到满充/放需要 4–7 步即 4–7 小时），具体值在 pilot run 中通过对比 0.1 / 0.2 / 0.3 的训练曲线确定。

### 3.2 观测空间

TES 模块新增 3 维观测：

| 维度 | 名称 | 范围 | 说明 |
|------|------|------|------|
| 1 | TES_SOC | [0, 1] | 归一化蓄冷量（0=空罐, 1=满罐） |
| 2 | TES_avg_temp | 归一化 | 罐内平均温度，反映冷量品质 |
| 3 | TES_valve_position | [-1, 1] | 当前阀门开度，增量控制所必需 |

不额外添加 chiller_PLR 和 CHW_supply_temp，因为这两个信息已包含在原始 27 维中的 `act_Chiller_T`（冷机温度设定状态，与 PLR 高度相关）和 `CW_temperature`（冷冻水侧温度）。

## 4. PV 建模

### 4.1 设计原则：离线预计算 + 时间序列注入

PV 出力作为外生时间序列注入 RL 环境的观测空间，agent 无需学习气象变量到 PV 出力的物理映射。理由：

1. **关注点分离**：论文核心是 reward shaping 对调度方向的影响，PV 出力是外部驱动信号而非优化对象
2. **避免冗余学习**：气象→PV 出力的映射已被 PVGIS 用确定性物理模型精确描述（6 参数效率模型），让 agent 用神经网络重学已知公式是对网络容量的浪费
3. **文献先例**：RL 领域将不可控外部信号视为 exogenous data 是标准做法（Ju & Crozier 2024 将 LMP 和太阳能出力直接注入 Gymnasium 环境）

### 4.2 数据来源与生成方法

使用 PVGIS 5.2（欧盟联合研究中心）或 Python pvlib 库的 `get_pvgis_hourly()` 函数，指定 Singapore 坐标、6 MWp 装机容量、c-Si 组件，一次性生成全年逐时 PV 出力时间序列。PVGIS 内部完成辐射→倾斜面辐射→组件温度（Faiman 模型）→PV 功率的全部计算链。

### 4.3 注入观测的 PV 信号（3 维）

| 维度 | 名称 | 说明 |
|------|------|------|
| 1 | current_pv_ratio | 当前 PV 出力 / DC 峰值负荷（归一化） |
| 2 | pv_future_slope | 未来数小时 PV 出力趋势斜率 |
| 3 | time_to_pv_peak | 到 PV 日内峰值的归一化时间 |

全部从预计算时间序列中实时查表得到。

## 5. 电价信号

### 5.1 数据来源

使用 Singapore National Electricity Market（NEMS）的 Uniform Singapore Energy Price（USEP）。USEP 是每半小时结算的批发电价，可从 EMC 网站（emcsg.com）免费下载 CSV 格式的历史数据（五年滚动）。

下载 2023 年全年 USEP 半小时数据，聚合为小时均值，形成 365 天 × 24 小时的电价矩阵。在 RL 训练中按"典型年"方式循环使用，与 TMY 气象数据的处理方式一致。Singapore 电价主要由天然气成本和系统负荷驱动，与当日天气的直接相关性弱，天气年和电价年的错位对结论影响小。

### 5.2 注入观测的电价信号（3 维）

| 维度 | 名称 | 说明 |
|------|------|------|
| 1 | current_price | 当前时刻的归一化电价 |
| 2 | price_future_slope | 未来数小时的电价趋势斜率 |
| 3 | price_mean | 未来数小时的电价均值 |

从预计算电价矩阵中实时查表得到。

### 5.3 RL-Green 的虚拟绿电价格

RL-Green 实验中，reward 使用虚拟绿电价格 $c_{\text{eff}} = \min(c_{\text{USEP}}, c_{\text{pv}})$，在 PV 发电时段 $c_{\text{pv}} \approx 0$，引导 agent 将负荷移向太阳能高峰时段。虚拟价格仅出现在 reward 计算中，不进入观测空间——agent 的观测中只看到真实电价和 PV 出力。
